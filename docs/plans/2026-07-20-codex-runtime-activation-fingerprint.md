# Codex Runtime Activation Fingerprint and Same-Task Restart Implementation Plan

> **For Codex:** REQUIRED SUB-SKILL: Use `ironclaude:executing-plans` to implement this plan task-by-task.

**Goal:** Make IronClaude self-updates verifiably active after a full Codex restart while preserving the same native task, using provider-specific startup fingerprints and a behavioral transition proof.

**Requirements:** `docs/plans/2026-07-19-multi-provider-v1-1-requirements.md` — implement MP-W12 and preserve MP-W08 and MP-W11.

**Design:** `docs/plans/2026-07-20-codex-runtime-activation-fingerprint-design.md`

**Architecture:** State-manager captures an immutable provider-specific manifest/bundle fingerprint before accepting MCP requests and passes it into existing `run_diagnostics`. Self-update guidance requires cachebuster-before-final-build, full Codex quit/relaunch, reopening the same native task, exact runtime/cache comparison, and a normal different-stage transition returning `changed:true`; a new task is recovery-only.

**Tech stack:** TypeScript 5.9, Node.js crypto/fs/path/url, Vitest, better-sqlite3, pytest contract tests, Codex plugin CLI, plugin-creator cachebuster/validator scripts.

**Execution mode:** Inline. Task 3 crosses a full Codex application restart; orchestration, task identity verification, MCP calls, review invocation, and continuation must remain in the main task.

**No-commit boundary:** Stage only explicit plan files. Do not commit, squash, or push.

```ts
export type RuntimeFingerprint = {
  plugin_name: string;
  plugin_version: string;
  plugin_root: string;
  manifest_path: string;
  manifest_sha256: string;
  bundle_path: string;
  bundle_sha256: string;
  client: IronClaudeClient;
};

export type RuntimeFingerprintCapture =
  | { ok: true; runtime: RuntimeFingerprint }
  | { ok: false; error: string };

export type RuntimeFingerprintExpectation = Pick<
  RuntimeFingerprint,
  'plugin_version' | 'plugin_root' | 'manifest_sha256' | 'bundle_sha256' | 'client'
>;

export type RuntimeActivationVerification =
  | { ok: true }
  | { ok: false; errors: string[] };

export function captureRuntimeFingerprintFromPaths(
  pluginRoot: string,
  bundlePath: string,
  client: IronClaudeClient,
): RuntimeFingerprintCapture;

export function captureRuntimeFingerprint(
  moduleUrl: string,
  client: IronClaudeClient,
): RuntimeFingerprintCapture;

export function verifyRuntimeActivation(
  capture: RuntimeFingerprintCapture | undefined,
  expected: RuntimeFingerprintExpectation,
): RuntimeActivationVerification;
```

Required cases:

1. `client='codex'` selects `.codex-plugin/plugin.json`; `client='claude'` selects `.claude-plugin/plugin.json`.
2. Both success cases assert exact absolute paths, parsed version, client, and SHA-256 computed from fixture bytes.
3. Capture fixture A, overwrite both manifest and bundle with fixture B, and assert the captured object remains byte-for-byte A.
4. Each provider fails when its active manifest is missing even when the other provider manifest is valid.
5. Malformed JSON, wrong plugin name, empty version, missing bundle, and directory/unreadable input return `{ok:false}` with actionable errors. Unsupported client coverage remains in `session-identity.test.ts` because `parseIronClaudeClient` rejects it before capture.
6. `verifyRuntimeActivation` rejects missing/failed capture and each independently mismatched expected field, while an exact match returns `{ok:true}`.

### Step 1.3: Implement provider-specific immutable capture (GREEN)

Create `worker/mcp-servers/state-manager/src/runtime-fingerprint.ts` using `createHash('sha256')`, `readFileSync`, `fileURLToPath`, and path resolution. Implement these exact invariants:

```ts
const MANIFEST_BY_CLIENT: Record<IronClaudeClient, string> = {
  codex: '.codex-plugin/plugin.json',
  claude: '.claude-plugin/plugin.json',
};

function sha256(bytes: Buffer): string {
  return createHash('sha256').update(bytes).digest('hex');
}
```

- `captureRuntimeFingerprint(import.meta.url, client)` treats the module file as `mcp-servers/state-manager/dist/index.js` and walks exactly three parents to plugin root.
